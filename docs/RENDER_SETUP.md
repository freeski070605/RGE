# RGE Render Setup

This guide is for deploying `RGE` to Render with:

- `rge-api` as the web service
- `rge-workers` as the background worker service
- `rge-cache` as the Render Key Value instance
- MongoDB hosted outside Render, such as MongoDB Atlas
- Cloudinary for uploaded assets and generated media

The repo already includes a Render Blueprint at [render.yaml](/e:/code/ReemTeamMasterWeb/RGE/render.yaml).

Important:
This guide assumes the Git repository you connected to Render is the `RGE` app itself.
That means the Render repo root is already the app root, so `Root Directory` should be left blank.

## Architecture

RGE is deployed as two app services plus one queue datastore:

- `rge-api`
  Serves the dashboard and API.
- `rge-workers`
  Runs BullMQ workers for intelligence sync, content generation, media generation, and scheduled publishing.
- `rge-cache`
  Render Key Value used by BullMQ.

MongoDB is external.
Cloudinary is external.
Instagram credentials are external.

## Before You Start

You need these accounts and credentials ready:

- Render account and Git repo connected to Render
- MongoDB Atlas connection string
- Cloudinary `cloud name`, `api key`, and `api secret`
- Instagram business publishing credentials
- ReemTeam backend feed URL and internal token

You also need one operator login for the dashboard:

- `OPERATOR_EMAIL`
- `OPERATOR_PASSWORD`

## Recommended Order

1. Push the latest `RGE` changes to your Git provider.
2. Verify `RGE/render.yaml` is committed.
3. Create the MongoDB Atlas database and allow Render network access.
4. Create the Render Blueprint from the repo.
5. Fill in the required environment variables.
6. Deploy.
7. Run the post-deploy smoke test.

## Create The Render Blueprint

In Render:

1. Click `New`.
2. Choose `Blueprint`.
3. Connect the repo that contains `RGE`.
4. Point Render at the repo root that contains [render.yaml](/e:/code/ReemTeamMasterWeb/RGE/render.yaml).
5. Leave `Root Directory` blank.
5. Review the three resources Render will create:
   - `rge-api`
   - `rge-workers`
   - `rge-cache`
6. Create the Blueprint.

Notes:

- Do not set `Root Directory` to `RGE` unless your repo actually contains a nested `RGE/` folder.
- The Blueprint already wires `REDIS_URL` from `rge-cache`.
- `JWT_SECRET` and `RGE_INTERNAL_TOKEN` are auto-generated for `rge-api`.
- `rge-workers` inherits shared secrets from `rge-api` through the Blueprint.

## Required Environment Variables

Set these on `rge-api` in Render.
The worker inherits the shared ones from the Blueprint.

### Required

- `MONGODB_URI`
  Your MongoDB Atlas connection string.
- `APP_BASE_URL`
  The public HTTPS URL of `rge-api`, for example `https://rge-api.onrender.com` or your custom domain.
- `OPERATOR_EMAIL`
  Dashboard login email.
- `OPERATOR_PASSWORD`
  Dashboard login password.
- `CLOUDINARY_CLOUD_NAME`
- `CLOUDINARY_API_KEY`
- `CLOUDINARY_API_SECRET`
- `BACKEND_API_BASE_URL`
  Public or internal base URL for the ReemTeam backend that exposes `/api/rge/feed`.
- `BACKEND_INTERNAL_TOKEN`
  Token RGE sends to the ReemTeam backend.
- `INSTAGRAM_USER_ID`
- `INSTAGRAM_ACCESS_TOKEN`

### Usually Keep As-Is

- `NODE_ENV=production`
- `CLOUDINARY_FOLDER=rge`
- `INSTAGRAM_GRAPH_VERSION=v23.0`
- `FFMPEG_PATH=ffmpeg`

### Auto-Generated By Blueprint

- `JWT_SECRET`
- `RGE_INTERNAL_TOKEN`

Do not replace these unless you intentionally want to rotate them.

## MongoDB Atlas Setup

RGE uses MongoDB, so you should provision Atlas before deploying.

1. Create a new Atlas cluster.
2. Create an application database user.
3. Copy the connection string into `MONGODB_URI`.
4. In Atlas Network Access, allow connections from Render.
   Best option: allow Render egress/static IPs only if your plan supports that.
   Fallback option: `0.0.0.0/0` temporarily while validating deployment.
5. Confirm the connection string includes the database name you want RGE to use.

## Cloudinary Setup

RGE no longer relies on shared local disk between services.
Uploads, edited assets, and generated media are stored in Cloudinary.

Use the values from your Cloudinary console:

- `CLOUDINARY_CLOUD_NAME`
- `CLOUDINARY_API_KEY`
- `CLOUDINARY_API_SECRET`

Optional:

- `CLOUDINARY_FOLDER`
  Defaults to `rge`. Change this only if you want a different Cloudinary folder prefix.

## Instagram Setup

RGE publishes Instagram content through the Instagram Graph API.

You need:

- An Instagram professional account
- A connected Facebook Page
- A valid long-lived access token
- The IG user id for the publishing account

Set:

- `INSTAGRAM_USER_ID`
- `INSTAGRAM_ACCESS_TOKEN`

Current implementation supports:

- `instagram`
- `story`

## ReemTeam Backend Feed Setup

RGE intelligence sync calls:

- `{BACKEND_API_BASE_URL}/api/rge/feed?days=<n>`

Make sure the ReemTeam backend:

- exposes that route in production
- accepts `x-rge-token`
- validates `BACKEND_INTERNAL_TOKEN`

## Deploy

After all required env vars are filled:

1. Trigger a deploy of `rge-api`.
2. Trigger a deploy of `rge-workers`.
3. Wait for both services to become healthy.

Expected results:

- `rge-api` passes `GET /api/health`
- `rge-workers` stays running without crash loops
- `rge-cache` shows as available

## Post-Deploy Smoke Test

Use this exact sequence:

1. Open the `rge-api` URL.
2. Sign in with `OPERATOR_EMAIL` and `OPERATOR_PASSWORD`.
3. Confirm the dashboard loads.
4. Upload one image asset.
5. Run `Auto-edit` on that asset.
6. Trigger `Sync backend feed`.
7. Create a brief from a content idea.
8. Generate variants.
9. Create media for a variant.
10. Confirm the variant shows Cloudinary-hosted media.
11. Schedule a publishing job.
12. Publish a variant immediately.
13. Confirm:
    - publishing job status updates
    - provider response is stored
    - permalink is present when returned by provider

## Troubleshooting

### Web deploy succeeds but media fails

Check:

- `CLOUDINARY_*` variables
- `FFMPEG_PATH`
- worker logs in `rge-workers`

### Dashboard says unauthorized

Check:

- `OPERATOR_EMAIL`
- `OPERATOR_PASSWORD`
- browser cookies
- `APP_BASE_URL` is correct and using HTTPS

### Intelligence sync fails

Check:

- `BACKEND_API_BASE_URL`
- `BACKEND_INTERNAL_TOKEN`
- ReemTeam backend logs

### Worker crashes on startup

Check:

- `MONGODB_URI`
- `REDIS_URL`
- inherited env vars in Render
- build logs and runtime logs for `rge-workers`

### Instagram publish fails

Check:

- `INSTAGRAM_USER_ID`
- `INSTAGRAM_ACCESS_TOKEN`
- account type and permissions
- whether the asset type is allowed for the chosen publish mode
